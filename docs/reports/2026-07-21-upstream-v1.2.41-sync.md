### 4.2 离线逐文件验证（Windows/macOS）

```python
import hashlib
import json
from pathlib import Path, PurePosixPath

lock = json.loads(Path("upstream/source-lock.json").read_text(encoding="utf-8"))
root = Path(lock["snapshot"]["path"])
expected = lock["files"]
actual_paths = sorted(path.relative_to(root).as_posix() for path in root.rglob("*") if path.is_file())
assert actual_paths == [item["path"] for item in expected]
manifest = bytearray()
total_bytes = 0
for item in expected:
    data = (root / PurePosixPath(item["path"])).read_bytes()
    blob = hashlib.sha1(b"blob " + str(len(data)).encode("ascii") + b"\0" + data).hexdigest()
    sha256 = hashlib.sha256(data).hexdigest()
    assert len(data) == item["size"]
    assert blob == item["git_blob_sha1"]
    assert sha256 == item["sha256"]
    manifest.extend(f"{sha256}  {item['path']}\n".encode("utf-8"))
    total_bytes += len(data)
assert total_bytes == lock["manifest"]["total_bytes"]
assert hashlib.sha256(manifest).hexdigest() == lock["manifest"]["sha256"]
assert not (root / ".git").exists()
print("UPSTREAM_SNAPSHOT_VERIFY_OK")
```

## 五、排错与根因证据

### Windows `tar.exe` 无法解压中文路径

- 现象：Windows 自带 `tar.exe` 在两个中文 Markdown 路径处报告 `Invalid empty pathname`，并把 UTF-8 文件名显示为乱码。
- 根因：归档本身包含 `348` 个合法成员，其中 `8` 个路径含中文；Python 3.13 `tarfile` 能无替换字符读取全部成员。归档只含 `71` 个目录和 `277` 个普通文件，无符号链接、硬链接或路径穿越。
- 处理：安全检查后使用 Python `tarfile` 按字节解压，再逐文件计算 Git blob SHA-1，与正确递归 tree 的 `277` 个 blob 全量对账。
- 结论：失败来自 Windows `tar.exe` 的 Unicode 路径处理，不是 Release 归档损坏；不得通过改名、漏文件或选择性解压规避。

### Git Trees API 根 SHA 使用错误

- 现象：最初把提交 SHA 传给 Git Trees API，递归条目可返回，但响应根 SHA 不等于提交记录中的 tree SHA；锁定生成器拒绝继续。
- 根因：提交对象与 tree 对象是不同 Git 对象；完整性锁定必须先读取 `commit.tree.sha`，再以该 tree SHA 查询递归树。
- 处理：废弃旧树证据，使用 `22e3a9c8ad15e18b972eae44a892b7980dca5ec2` 重新读取未截断递归树，并对临时快照和工作区快照各执行一次完整 blob 对账。
- 结论：两处快照均得到相同清单摘要 `3c9b16802f49a1bcb56fda9630d97edc52c918c30d1924145244d9239801d3d4`。

### 上游 `.gitignore` 忽略审计文件

- 现象：首次普通暂存只有 `261` 条路径，少于快照 `277` 个文件、锁定文件和报告合计应有的 `279` 条路径。
- 根因：其中 `8` 条是 Git 对中文路径的转义显示，不是缺失；其余 `18` 个 `docs/superpowers/` 文件被上游自身 `.gitignore` 规则忽略。
- 处理：不修改上游 `.gitignore`，仅对该目录使用 `git add -f`，然后用 `git diff --cached --name-only -z` 按 UTF-8 原始路径重新比对。
- 结论：暂存区最终包含全部 `279` 条预期路径，不增加任何快照外文件。

### Windows 暂存未保留可执行 mode

- 现象：`scripts/installer/macos/package-dmg.sh` 内容 blob 正确，但首次暂存 mode 为 `100644`，上游 mode 为 `100755`。
- 根因：Windows 文件系统和 Git 默认配置不从工作区可执行位恢复上游 mode。
- 处理：使用 `git update-index --chmod=+x` 只修正该路径的索引 mode。
- 结论：暂存索引已恢复为 `100755 287778ee2d16ac56ed6bf02b54283df5ab03c347`，其余 `276` 个快照文件 mode/blob 均与上游一致。
### 上游原始 whitespace 不做清洗

- 现象：`git diff --cached --check` 报告上游 `CHANGELOG.md`、YAML、Markdown 和资源文件中的原始尾随空格或文件末尾空行。
- 根因：这些字节来自上游 Release，且已通过每个文件的 Git blob SHA-1 与 SHA-256 对账；它们不是本次同步新增的格式错误。
- 处理：不修改上游文件；同步 PR 对新增的 `source-lock.json` 与本报告执行 `git diff origin/main...HEAD --check -- ...`，对 `upstream/CodexPlusPlus/` 使用 source-lock blob 对账替代格式化检查。
- 结论：保留原始字节优先于格式清洗，避免把审计快照变成未经批准的内容改写。

### Git 默认路径转义导致允许路径误判

- 现象：Windows 默认 `core.quotePath=true` 时，PR 差异中的 `8` 个中文路径被 Git 以八进制转义并包裹引号，允许路径正则会把它们误判为快照外文件。
- 根因：路径范围验证依赖了调用机器的 Git 全局输出配置，没有固定 UTF-8 路径输出。
- 处理：将 PR 路径枚举固定为 `git -c core.quotePath=false diff --name-only origin/main...HEAD`；不改动快照文件或全局 Git 配置。
- 结论：默认配置复现 `8` 个误判，固定 `core.quotePath=false` 后 `279` 条变更路径全部落在 `278` 条 `upstream/` 与 `1` 条同步报告内。
