# inputcodex 项目总计划

## 状态

- 当前阶段：筹备与发现。
- 当前结论：只完成仓库与治理基线，尚未批准源码导入或架构实施。
- 跟踪入口：GitHub Issue `#1`。

## 不变量

- 产品名称为 `inputcodex`。
- 产品无广告、无推广位、无广告 SDK。
- 性能与稳定性必须有可重复测量证据。
- 功能加载必须可取消、可隔离、可诊断。
- 架构选择以审计结果和可验证原型为依据，不以原项目结构为默认答案。

## 阶段门

### Gate 0：仓库准备

- 建立公开仓库、许可证与项目文档。
- 建立 Issue、会话计划和运行工作流。
- 不导入源码。

### Gate 1：双仓审计

- 盘点技术栈、模块、依赖、许可证与构建链。
- 定位广告、遥测、卡顿和加载异常相关实现。
- 建立可复现的行为与性能基线。

### Gate 2：方案决策

- 对比从零重写、渐进重构和新壳迁移三种路线。
- 明确模块边界、状态模型、并发模型和错误隔离策略。
- 定义首个可验收版本、性能预算和迁移边界。
- 获得用户明确批准。

### Gate 3：最小垂直切片

- 只实现一个可运行、可测试、可测量的核心闭环。
- 不引入广告与非必要后台行为。
- 用测试和性能数据验证架构假设。

### Gate 4：分批迁移与发布

- 按模块迁移功能，每批独立评审与回归。
- 保持构建文档、排错记录和发布证据同步。
- 达到发布门槛后再确定稳定版本策略。

## 当前停止条件

满足以下任一条件时停止实施并返回讨论：

- 参考仓库许可证或来源无法确认。
- 无法复现用户所述卡顿或功能加载异常。
- 技术路线发生变化但尚未获得用户批准。
- 需要实现 UI，而用户尚未明确授权当前助手承担 UI 工作。
