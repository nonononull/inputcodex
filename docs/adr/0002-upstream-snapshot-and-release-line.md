# 缓存完整上游快照并维护自主发布线

`inputcodex` 从 `BigPizzaV3/CodexPlusPlus v1.2.41` 的完整源码快照开始，不导入上游历史提交。上游快照进入本仓库用于功能审计和重构追踪，但不直接参与新产品构建；版本采用 `v<上游版本>-inputcodex.<修订号>`，所有安装包、更新清单、签名和下载地址均由 `nonononull/inputcodex` 发布。

## 备选方案

- Git Submodule：只保存外部指针，源码没有完整进入本仓库治理。
- 导入完整历史：审计能力最强，但显著增加仓库噪声和迁移复杂度。
- 只选择性复制文件：体积较小，但会丢失未迁移功能和副作用证据。

## 影响

- `upstream/CodexPlusPlus/` 只能由专门的上游同步 PR 修改。
- 上游同步 PR 与功能迁移 PR 必须分离。
- 上游中的 Tauri/React UI、现有注入脚本、远程推荐列表、广告代码和素材只保留为审计输入，不得直接进入新架构或最终运行面。
- 若需要保留这些资产背后的有效能力，必须通过独立功能或一致性例外 Issue 重新设计，不能复用其运行时实现或远程数据源。
- 客户端只信任本仓库的签名更新清单与 Release 资产。
