# inputcodex 项目语境

`inputcodex` 在保持上游有效功能行为一致的前提下，以纯 Rust 原生架构重新实现 Codex 桌面增强与管理能力。本文件只定义项目专用术语，不描述具体实现。

## 上游与基线

**上游稳定基线**：
`BigPizzaV3/CodexPlusPlus` 最新正式 Release，是功能一致性的唯一基准；当前起始基线为 `v1.2.41`、提交 `3dafffcafb2566a1e8bce4b35671656d6adb3eda`。
避免用语：上游最新版、参考版本、原版

**上游预警源**：
`BigPizzaV3/CodexPlusPlus` 的 `main` 分支，只用于提前发现变化，不直接改变功能基线。
避免用语：开发基线、自动同步源

**上游快照**：
保存在 `inputcodex` 仓库内的完整上游源码副本，不包含上游 Git 历史；只能通过上游同步 PR 更新。
避免用语：Submodule、依赖源码、Vendor 依赖

**半成品参考**：
`zsr131550/CodexPlusPlus`，仅提供原生迁移、功能合同、测试与性能排错证据，不是功能真源或代码底座。
避免用语：下游基线、新架构基线

**审计专用上游资产**：
上游 Tauri/React 管理界面、现有注入脚本、远程推荐列表、广告代码和素材；它们随完整快照进入仓库，但不得作为新架构代码、运行时依赖或动态数据源。其背后的有效能力需要独立功能或一致性例外 Issue 才能重新设计。
避免用语：待迁移模块、运行时资源、兼容依赖

## 一致性与差异

**功能一致性**：
输入、输出、数据兼容、持久化结果、副作用、错误语义和操作流程与上游稳定基线一致；不要求 UI 像素一致。
避免用语：代码一致、界面复制、文件一致

**一致性例外**：
因功能无效、有害副作用、错误语义争议、广告、推广或遥测而不能直接保持一致的行为；必须由项目所有者在 Issue 中决定。
避免用语：顺手修复、合理优化、默认删除

**迁移项**：
把一个已登记的上游功能行为重新实现到 `inputcodex` 新架构中的有界交付单元。
避免用语：搬代码、同步功能

## 版本与发布

**上游基础版本**：
`inputcodex` 当前对齐的上游稳定版本，例如 `1.2.41`。
避免用语：本地版本、兼容版本

**inputcodex 修订号**：
在同一上游基础版本上的独立发布序号，例如 `inputcodex.1`、`inputcodex.2`。
避免用语：补丁号、内部构建号

**自主更新源**：
由 `nonononull/inputcodex` 提供的 Release、更新清单、签名和下载资产；客户端不得从上游仓库下载更新。
避免用语：上游更新源、镜像更新源
